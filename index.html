TFT Set 17, By The Numbers

Samuel Bingham · SQL Analysis · July 2026

A SQL analysis of my own Set 17 ranked match history--556 games, pulled from Riot's API into a relational database and interrogated the way I'd want a PM to interrogate product data: pose a hypothesis, run the query, and be honest about what the number actually supports.

Project Overview

I think the natural extension of the product teardown I wrote on TFT is to turn the same scrutiny on my own play. If I'm going to argue that Riot should build post-game analysis tooling for mid-ranked players, I should be willing to run that kind of analysis on myself first--with real data, not just informed feel.

This project pulls my full Set 17 ranked match history through Riot's TFT API, models it into a relational schema, and asks it the same kind of questions I'd ask of any product dataset. Pose a hypothesis, run the query, and be honest about what the result actually supports--including the times it didn't support what I expected going in.

Set 17 -- 720 LP peak across 561 games. Grandmaster (394 LP) as of this writing.

I fetched 1,000 games total from the API, though only 563 of those turned out to be Set 17--the rest was leftover history from Set 16 sitting on the same account. Of the 563, I ended up excluding 7 as outlier games that don't reflect how I actually play; more on how and why in the next section. That left 556 games behind every number on this page.

Why this, and why SQL: the point was never to build the most sophisticated pipeline I could. It was to practice real SQL--joins, aggregation, a hand-rolled correlation coefficient--against a dataset I actually have the context to sanity-check. A few of the findings below didn't survive that sanity check on the first pass. I've left that process visible below rather than smoothing it over.

Building the Dataset

Getting from raw match history to something queryable took four small scripts and one decision I didn't expect to have to make going in: how to define a game that shouldn't count.

The Pipeline

Four pieces do the actual work:

src/fetch_matches.py   Riot API -> cached raw match JSON (data/raw/) src/schema.sql         normalized SQLite schema + analysis views src/load_db.py          parses cached JSON -> data/tft.db analysis/*.sql          the actual analysis queries, one file per question

Five tables hold the data--matches, participants, units, unit_items, and traits--plus two views that do the scoping so I'm not repeating the same WHERE clause in every query. set17_participants filters everything down to Set 17; set17_participants_clean additionally strips the outlier games below. Augments were part of the original plan and had to be cut entirely--they simply aren't present anywhere in Riot's current match-v1 response, which was itself a small finding I didn't go looking for.

The Games That Almost Slipped Through

Two of my own habits show up in this data clearly enough that, left in, they would have quietly skewed every trait and comp number on this page.

The first is selling my board to hyper-roll for a three-star five-cost, something I do occasionally when a lobby is already lost and I'd rather gamble for a strong finish than accept a guaranteed bad one. It leaves a distinct signature: an abnormally low final unit count--there's a real gap in the data, games sit at three or four units or jump straight to six or more, nothing in between--paired with gold_left near zero and a very late last_round.

The second is conceding once eighth place is already locked in. That one turned out to be rarer than I expected, and showed up as exactly one game: 147 gold left at round 20, with a level too low for how far the game had progressed to be anything but someone who'd stopped trying.

I found this out the hard way: a full 9-11 unit board running three copies of a five-cost is just a strong, legitimate comp, not a reroll gamble. The exclusion rule needed the shrunk board and the concentrated five-cost signal together--either one alone produced false positives.

In the end, only 7 of 563 games--about 1 in 80--met both conditions and were cut.

Economy vs. Placement

Does gold left at game-end actually predict placement?

Pearson r = -0.013 (n=556)

On its own, that number says almost nothing--essentially zero linear relationship. But the shape underneath it isn't flat, it's a shallow U, and that shape is more informative than the single coefficient that's supposed to summarize it. Both hoarding gold (31+) and burning all the way down to zero correlate with worse results than sitting in the 1-30 middle range. The correlation coefficient alone would have told me to stop looking.

The zero-gold bucket turned out to be two different games wearing the same number. Splitting it by how far the game had progressed, the five times I hit zero gold before round 25--despite a perfectly reasonable level--all ended in eighth place. Zero top-four finishes, no exceptions. That's about as clean a "forced all-in that failed" signal as this dataset produces. The other 118 zero-gold games, the ones that happened later, are far more mixed at 44.1% top-four. And the detail I didn't expect going in: none of my 82 wins ever end with zero gold left--winning games keep money in the bank. Zero gold isn't one pattern, it's a forced-spend signal, and whether it reads as a comeback or a death spiral depends entirely on when it happens.

The Shen comp--Assassin, Astronaut, Fateweaver, Timebreaker--is clearly my best line by every measure that matters: 3.08 average placement, 84.6% top-four, and a 38.5% win rate across thirteen games. That's the one I should be prioritizing, and mostly haven't been.

The more interesting result is at the bottom, and it isn't simply "worst comp." The Fiora/Tahm Kench/DRX/PsyOps/HP Tank line hits top-four more than half the time (58.3%) across twelve games, but has never once converted to a win. That's not a weak comp, it's a comp with a ceiling--good for stabilizing a lobby that's already going sideways, bad if the plan is to actually close a game out.

This groups by exact trait-set match, so two games that are functionally the same comp with one flex-slot trait swapped won't merge here. A simplification I made on purpose, not something I missed.

Performance Trend Over Time

I went into this expecting a climbing line, and I don't think the data supports one. Week 15, right after Set 17 launched, was my strongest stretch by a wide margin--3.63 average placement, 68.6% top-four--and nothing since has matched it. Results settled into a flat plateau around 4.2 to 4.6 for the following three months, with no real upward trend to point to.

I'd rather report that honestly than force a narrative the numbers don't back. It does leave a real open question, though: was week 15 a strong comp I later lost access to as the meta shifted, or is thirty-five games just too small a sample to trust? I don't have enough here to answer that yet, but it's exactly the kind of question a follow-up pass at this dataset should try to close.

Damage Output vs. Placement (and Why It's a Trap)

Does dealing more damage to other players correlate with placement--and is that actually useful to know?

Pearson r = -0.916 (n=556)

By far the strongest correlation in the entire analysis, and the one I trust the least.

I'm including this one on purpose, not because it's a real finding but because it's a useful trap to walk through. total_damage_to_players accumulates every round a board is still alive, so surviving longer--which is, definitionally, what placement measures--mechanically produces more damage dealt, independent of whether the comp underneath it is actually good. A correlation this strong should make you suspicious, not confident. Trait choice, econ discipline, and comp selection are decisions I can act on next game. Damage dealt is just placement, wearing a different unit.

Limitations

The honest counterargument to all of this is that it's a sample size of one--556 games from a single account, in a single meta, played by someone who already knows this game well enough to bias his own reading of the results.

That's fair, and I don't think there's a clean way around it inside the scope of this project. What I can say is that the findings above are internally consistent--the trait, comp, and econ numbers all point in compatible directions rather than contradicting each other--which is weak evidence they're capturing something real about how I play, even if they say nothing about how anyone else should.

A few other honest gaps: augment data isn't in the API response I have access to, so that analysis had to be cut entirely rather than approximated. Comp identification uses exact trait-set matching, not fuzzy clustering, so minor flex-slot variations don't merge into the same row. And the damage-correlation finding above is included as a methodological example, not a real one--worth restating here so it doesn't get read as a fifth genuine insight.

A note on perspective: I've played this game long enough, and this account well enough, that I'm not a neutral observer of my own data. Where a result confirmed something I already believed, I tried to hold it to the same bar as the ones that surprised me. I can't promise I always succeeded.

Anyways, if you made it this far--thank you for reading. The full pipeline, schema, and every query behind these numbers are in the GitHub repo, if you want to see exactly how the sausage got made. And if you haven't already, the other half of this project is my product teardown of TFT as a live service--same game, opposite direction. That one argues for what Riot should build. This one is me turning the same argument on myself.
