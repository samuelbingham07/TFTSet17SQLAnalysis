TFT Set 17, By The Numbers

Samuel Bingham · SQL Analysis · July 2026

An SQL analysis of my own Set 17 ranked match history--556 games, pulled from Riot's API into a relational database and interrogated the way I'd want a PM to interrogate product data. I pose a hypothesis, run the query, and analyze what the number might support.

Project Overview

I think the natural extension of the product teardown I wrote on TFT is to turn the same scrutiny on my own play. If I'm going to argue that Riot should build post-game analysis tooling for mid-ranked players, I should be willing to run that kind of analysis on myself first, and with real data.

This project pulls my full Set 17 ranked match history through Riot's TFT API, models it into a relational schema, and asks it the same kind of questions I'd ask of any product dataset. Pose a hypothesis, run the query, and be honest about what the result actually supports--including the times it didn't support what I expected going in.

Set 17 -- 720 LP peak across 561 games. Grandmaster (394 LP) as of this writing.

I fetched 563 final boards from the API. Of the 563, I ended up excluding 7 as outlier games whose final boards don't reflect the games themselves at all. more on how and why in the next section. That left 556 games behind every number on this page.

Why this, and why SQL: the point was never to build the most sophisticated pipeline I could. It was to practice real SQL (joins, aggregation, a hand-rolled correlation coefficient) against a dataset I actually have the context to sanity-check. A few of the findings below didn't survive that sanity check on the first pass, and I've left those visible.

Building the Dataset

Getting from raw match history to something queryable took four small scripts and the question how to define a game that is not representative of my play and should be omitted.

The Pipeline

Four pieces are at work:

src/fetch_matches.py   Riot API -> cached raw match JSON (data/raw/) src/schema.sql         normalized SQLite schema + analysis views src/load_db.py          parses cached JSON -> data/tft.db analysis/*.sql          the actual analysis queries, one file per question

Five tables hold the data--matches, participants, units, unit_items, and traits--plus two views that do the scoping so I'm not repeating the same WHERE clause in every query. set17_participants filters everything down to Set 17; set17_participants_clean additionally strips the outlier games below. Augments were part of the original plan and had to be cut entirely because they unfortunately aren't present anywhere in Riot's current match-v1 response (for good reason, since I think augment stats abuse does ruin player freedom and decision-making).

The Games That Almost Slipped Through

Two of my own habits show up in this data clearly enough that, left in, they would have quietly skewed every trait and comp number on this page.

The first is selling my board to hyper-roll for a three-star five-cost, something that occasionally becomes the optimal play when a lobby is already lost unless I gamble for a strong finish than accept a guaranteed 3rd or 2nd. In these games, there's a real gap in the data, final boards are comprised of just 5 costs with no syngeries (since all other units were sold of for rerolling), and paired with gold_left near zero and a very late last_round.

The second is leaving the game (like a ragequit) once eighth place is already locked in. That one turned out to be rarer than I expected, and showed up as exactly one game: 147 gold left at round 20, with a level too low for how far the game had progressed to be anything but someone who'd stopped trying.

In the end, only 7 of 563 games--about 1 in 80--met both conditions and were cut.

The Traits That Weren't Traits

A sharper-eyed reader than me caught this one: a handful of names sitting in the trait table--Mana, Stargazer (Wolf) among them--weren't traits I'd ever actually pick or see highlighted in my own trait bar. I went back into the raw data to check, and found eleven names (Mana, AP, AS, Assassin, Melee, Ranged, HP Tank, Shield Tank, Resist, Summon, Flex) that all share the same tell: a bare stat or role word instead of a real name, unlike every genuine trait in the set. Those are internal engine classification tags, not synergies I ever chose, and they were quietly riding along in the same API field as the real ones. I've excluded all eleven from both the trait performance and comp identification queries below--Finding 3 in particular reads differently now that comp signatures aren't padded with generic role tags alongside real traits.

02

Economy vs. Placement

Does gold left at game-end actually predict placement?

I grouped sub-10 gold as one group here rather than splitting out exactly-zero, because both represent the same underlying decision: capping out a board and spending down, knowing elimination might be close. What actually matters isn't 0 versus 7 gold--it's low gold versus a lot of banked gold, and whether that low-gold state shows up early or late.

Gold LeftGamesAvg PlacementTop4%
<104234.3952.2
10-19454.4951.1
20-29313.9454.8
30+574.4952.6

Pearson r = -0.013 (n=556)

That's essentially zero linear relationship, and once <10 is grouped as one bucket, the bucketed view is honestly pretty flat too--everything sits within half a placement of everything else, except a modestly better 20-29 bucket that's only 31 games and could easily be noise. Gold left at game-end, on its own, tells me almost nothing.

Splitting the low-gold bucket by how far the game had progressed is where the real signal shows up. The 18 times I hit under 10 gold before round 25--despite a perfectly reasonable level--produced an average placement of 7.83 with zero top-four finishes: fifteen eighths, three sevenths. That's a clean "capped out early and died anyway" pattern. The other 405 low-gold games, the ones that happened at round 25 or later, look completely different: 4.23 average placement, 54.6% top-four--basically average or slightly better. And the detail that actually surprised me: 60 of my 82 wins end with under 10 gold left. Once I checked, that stopped being surprising--a board that's already won doesn't need banked economy, so of course a lot of my best games end with an empty bank. Low gold isn't good or bad on its own. Early, it's a death spiral. Late, it's often just what winning looks like.

The Astronaut/Fateweaver/Shen/Timebreaker comp is clearly my best line: 3.07 average placement, 85.7% top-four, 35.7% win rate across fourteen games. That part isn't surprising. What is surprising is the bottom row: Astronaut/Fateweaver/Timebreaker--the exact same core, minus Shen--sits dead last at 5.91 average placement, 9.1% top-four, and zero wins in eleven games. One trait is the entire difference between my best comp and my worst. I hadn't clocked how load-bearing Shen was in that lineup until the data forced the two apart into separate rows.

The other result worth flagging: DRX/Fiora/Psionic/Tahm Kench hits top-four nearly three-quarters of the time (72.7%) across twenty-two games--my most-played comp on this list--but converts to a win only 4.5% of the time. That's not a weak comp, it's a comp with a ceiling: reliable enough to stabilize a lobby that's already going sideways, not built to actually close a game out.

This groups by exact trait-set match, so two games that are functionally the same comp with one flex-slot trait swapped won't merge here. A simplification I made on purpose, not something I missed.

I went into this expecting a climbing line, but I don't think the data supports one. Week 15, right after Set 17 launched, was my strongest stretch by a wide margin--3.63 average placement, 68.6% top-four--and nothing since has matched it. Results settled into a flat plateau around 4.2 to 4.6 for the following three months, with no real upward trend to point to.

I'd rather report that honestly than force a narrative the numbers don't back. It does leave a real open question, though: was week 15 a strong comp I later lost access to as the meta shifted, or is thirty-five games just too small a sample to trust? I don't have enough here to answer that yet, but it's exactly the kind of question a follow-up pass at this dataset should try to close.

Damage Output vs. Placement (and Why It's a Trap)

Does dealing more damage to other players correlate with placement--and is that actually useful to know?

Damage BucketGamesAvg PlacementTop4%
<50677.640.0

200+251.12100.0

Pearson r = -0.916 (n=556)

By far the strongest correlation in the entire analysis, and the one I trust the least.

I'm including this one on purpose, not because it's a real finding but because it's a useful trap to walk through. total_damage_to_players accumulates every round a board is still alive, so surviving longer--which is, definitionally, what placement measures--mechanically produces more damage dealt, independent of whether the comp underneath it is actually good. A correlation this strong should make you suspicious, not confident. Trait choice, econ discipline, and comp selection are decisions I can act on next game. Damage dealt is just placement, wearing a different unit.

Limitations

The honest counterargument to all of this is that it's a sample size of one--556 games from a single account, in a single meta, played by someone who already knows this game well enough to bias his own reading of the results.

That's fair, and I don't think there's a clean way around it inside the scope of this project. What I can say is that the findings above are internally consistent--the trait, comp, and econ numbers all point in compatible directions rather than contradicting each other--which is weak evidence they're capturing something real about how I play, even if they say nothing about how anyone else should.

A few other honest gaps: augment data isn't in the API response I have access to, so that analysis had to be cut entirely rather than approximated. Comp identification uses exact trait-set matching, not fuzzy clustering, so minor flex-slot variations don't merge into the same row. And the damage-correlation finding above is included as a methodological example, not a real one--worth restating here so it doesn't get read as a fifth genuine insight.

A note on perspective: I've played this game long enough, and this account well enough, that I'm not a neutral observer of my own data. Where a result confirmed something I already believed, I tried to hold it to the same bar as the ones that surprised me. I can't promise I always succeeded.

Anyways, if you made it this far--thank you for reading. The full pipeline, schema, and every query behind these numbers are in the GitHub repo, if you want to see exactly how the sausage got made. And if you haven't already, the other half of this project is my product teardown of TFT as a live service--same game, opposite direction. That one argues for what Riot should build. This one is me turning the same argument on myself.
