TFT Set 17, By The Numbers

Samuel Bingham · SQL Analysis · July 2026

A SQL-driven analysis of 556 ranked Teamfight Tactics games from my own Set 17 climb -- pulled from Riot's API into a normalized SQLite database and queried like a PM would query product data: pose a question, run it, interpret the result, and call out when a correlation isn't actually a lever.

Overview

This is a companion piece to my product teardown of TFT, but the goal here is different: instead of analyzing the game as a product, I'm analyzing my own play as data. I pulled my full Set 17 ranked match history through Riot's TFT API, modeled it into a relational schema, and ran it through the same kind of question-driven SQL analysis I'd want to see from an analytics partner on a live-service team.

Set 17 -- 720 LP peak across 561 games (Challenger). Currently Grandmaster at 394 LP.

1,000 games were fetched from the Riot API, of which 563 were Set 17 (the rest was leftover Set 16 history on the same account). Of those 563, 7 were excluded as outlier games that don't reflect normal play -- see the Data Cleaning section below for exactly how and why. That leaves 556 clean games behind every result on this page.

Why SQL, why this data: the point of this project wasn't to build the most sophisticated pipeline possible -- it's to practice real SQL (joins, window-adjacent aggregation, correlation math) against a dataset I have enough domain context to sanity-check. Every finding below got checked against "does this make sense given what I know about how the game is actually played" before it made the page. A few didn't survive that check, and where that happened, it's called out rather than smoothed over.

Methodology

Pipeline

src/fetch_matches.py   Riot API -> cached raw match JSON (data/raw/) src/schema.sql         normalized SQLite schema + analysis views src/load_db.py          parses cached JSON -> data/tft.db analysis/*.sql          the actual analysis queries, one file per question

Five tables (matches, participants, units, unit_items, traits), plus two views: set17_participants scopes everything to Set 17, and set17_participants_clean additionally excludes the outlier games below. Augment data was part of the original plan -- it isn't present anywhere in Riot's current match-v1 response, so that analysis had to be dropped.

Data Cleaning: Excluding Non-Representative Games

Two intentional playstyles show up in the data and would skew comp/trait performance if left in:

Selling the board to hyper-roll for a 3-star 5-cost, ending in 2nd/3rd. Signature: an abnormally low final unit count -- there's a clean gap in the data, games sit at 3-4 units or jump straight to 6+, nothing in between -- combined with gold_left near 0 and a very late last_round.

Conceding once 8th place was locked in. Signature: gold_left >= 100 at elimination, which occurs exactly once across all 563 games, combined with a level too low for how far the game had progressed.

A full 9-11 unit board running 3 copies of a 5-cost is a strong legitimate comp, not a reroll gamble. The exclusion rule needed the shrunk board and the concentrated 5-cost signal together -- either alone produced false positives.

Only 7 of 563 games (1.2%) met both conditions and were excluded.

Economy vs. Placement

Does gold left at game-end predict placement?

Gold LeftGamesAvg PlacementTop4%
01234.9142.3
1-103054.1955.7
11-20474.3255.3
21-30273.9655.6

Pearson r = -0.013 (n=556)

Essentially no linear relationship. But the shape isn't flat, it's a shallow U: both hoarding gold (31+) and hitting exactly 0 correlate with worse results than the 1-30 middle range. Digging into the 0-gold bucket specifically: splitting it by game stage shows the 5 games where gold hit 0 early (before round 25) despite a decent level all ended in 8th place -- 100%, 0% top-4, a clean "forced all-in that failed" signal. The other 118 late-game 0-gold games are far more mixed (44.1% top-4), and notably none of my 82 wins ever end with 0 gold left -- winning games keep gold banked. So "0 gold" isn't a single pattern, it's a forced-spend signal that's sometimes a stabilizing comeback and sometimes a death spiral.

Honest read: there's no clean "steady improvement" story here. Week 15, right after launch, was my strongest stretch by a wide margin, then results settled into a flat plateau around 4.2-4.6 for the following three months with no clear upward trend since. That's a more useful finding than a forced positive narrative -- it raises a real follow-up question (was week 15 a strong comp that later got weaker, or just small-sample variance?) rather than answering one.

05

Damage Output vs. Placement (and Why It's a Trap)

Does dealing more damage to other players correlate with placement -- and if so, is that actually useful to know?

Damage BucketGamesAvg PlacementTop4%
<50677.640.0
50-992095.9911.5
100-1491693.0892.3
150-199861.43100.0
200+251.12100.0

Pearson r = -0.916 (n=556)

By far the strongest correlation in the whole analysis -- and the least useful one.

total_damage_to_players accumulates every round you're still alive, so surviving longer, which is placement, mechanically produces more damage dealt, independent of whether the comp itself is actually strong. This is included deliberately as a contrast case: not every strong correlation is a lever you can pull. Trait choice, econ discipline, and comp selection are decisions; damage dealt is closer to a restatement of the outcome.

Limitations

  • Single-player dataset. 563 games from one account -- findings describe my results with these comps in this meta, not general win rates.
  • No augment data. It isn't present anywhere in the current Riot API response; the original plan included an augment-vs-placement analysis that had to be dropped.
  • Exact-match comp grouping. Comp identification uses exact trait-set matching, not fuzzy/clustered matching, so minor flex-slot variations don't merge.
  • Damage correlation is a methodological example, not a real finding -- included to show that a strong correlation isn't automatically an actionable one.
